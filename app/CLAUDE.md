## 1. What the app is

- **Connect!** ‚Äî a mobile-first pickup-sports matchmaking app for **Doha, Qatar** (currency **QAR**).
- Sports: **padel ¬∑ tennis ¬∑ badminton ¬∑ running**.
- Core user jobs: **discover** open matches ‚Üí **host** a match ‚Üí **join / request / approve** players (+ FIFO **waitlist**) ‚Üí **chat** (per-match group + DMs) ‚Üí **record results** post-match.
- Stage 1 scope: free matching only, cash-on-arrival (no in-app payments), 18+, Doha-only. See spec ¬ß1.

## 2. Tech & file structure

**Stack** ‚Äî React 19 + TypeScript, **Vite 8**, **Tailwind v4** (`@tailwindcss/vite`, config-less ‚Äî tokens live in `src/index.css`), `react-router-dom` v7, **lucide-react** icons, `clsx`/`tailwind-merge`/`cva` (via `cn()` in `lib/utils.ts`). No shadcn; components are local. Run: `npm run dev` (Vite), `npm run build` (`tsc -b && vite build`), `npm run lint`.

**Entry / frame** ‚Äî `index.html` (loads Inter + Instrument Serif + IBM Plex Sans Arabic) ‚Üí `main.tsx` ‚Üí `App.tsx` (`<Routes>`). Every screen renders inside `Shell` = the phone frame: `max-w-[430px]`, `h-dvh`, cream page on a dark backdrop, ambient `Blobs`, optional floating `BottomNav`.

**State** ‚Äî single **in-memory reactive store** (`lib/store.ts`) via `useSyncExternalStore`, seeded from `lib/mock/*`. Repository-shaped accessors mirror future Supabase queries. A few things persist to `localStorage`: `connect.freshAccount`, profile-done flag, and **`connect:hostedMatch`** (see ¬ß5). ES modules throughout ‚Äî **no `window` globals, no inline Babel** (that was the old prototype in `../Connect! UI Lab/`).

## 4. Data model

There are **two shapes**: the canonical store `Match` (Supabase-shaped, snake_case) and the **edit-form / `HostedMatch`** shape (camelCase, what the create/edit form + Home card use). Keep them in sync via the mapper in `EditMatchScreen.save()`.

**Store `Match`** (`lib/types.ts`) ‚Äî `id, host_id, sport, name?, venue_id?, venue_name, venue_location?, court_number?, route_start?, route_end?, round_trip, start_time(ISO), end_time(ISO), skill_min, skill_max, total_spots, spots_available, fee_total?, fee_per_player?, join_mode('open'|'approval'|'invite'), gender_restriction?('mixed'|'ladies'), status('active'|'cancelled'), notes?, created_at`. **`gender_restriction`** ('ladies' = female players only; absent ‚áí 'mixed') is **server-enforced** ‚Äî the gate lives in RLS + the join/request/invite/waitlist RPCs + DB triggers (see the changelog/¬ß5), NOT just the UI. The create/edit "Open to" toggle is shown ONLY to female hosts; `genderBlocks(d, matchId)` (store) mirrors the gate so the UI never offers a CTA the server will reject, and the `MatchCard genderBlocked` prop renders a disabled "Women only" state for a barred (male) viewer. The `LadiesBadge` (one component) renders on both card variants + Match Details. **Skill is a RANGE** over the 7-tier ladder (`skill_min`/`skill_max`, both `SkillTier`; `skill_min === skill_max` ‚áí a single level). DB CHECK enforces `skill_ord(skill_min) ‚â§ skill_ord(skill_max)`; `create_match` takes `p_skill_min`/`p_skill_max`. Read-time `MatchStatus` is computed, not stored (¬ß5). **`name`** is the optional host-given title from the create/edit form: persisted live via the `matches.name` column (`create_match`'s `p_name` + `updateMatch` cols + `repo.mapMatch`), and rendered as the **`MatchCard` title** when present (else the card falls back to the derived `matchKind at courtLabel`, e.g. "Doubles at 4"). So every `MatchCard` surface (Home "You're hosting", Discover, All Matches) shows the typed name; venue still shows on the card art + the MapPin line.

**`HostedMatch`** (`lib/hostedMatch.ts`) ‚Äî the host's own match, the form's working shape:

| Field | Meaning |
|---|---|
| `sport` | padel / tennis / badminton / running |
| `name` | match title (serif headline) |
| `dateKey` / `dateLabel` | ISO `yyyy-mm-dd` / display `"Thu ¬∑ May 23"` (derived from `dateKey` via `labelFromKey`, any month) |
| `startTime` / `endTime` | **24h `"HH:mm"`**, displayed as-is (no AM/PM anywhere ‚Äî see ¬ß5) |
| `matchType` | `casual` \| `competition` |
| `gender` | `mixed` \| `ladies` |
| `requireApproval` | join instantly vs approve (maps to `join_mode`) |
| `isFree` / `pricePerPlayer` | Free vs split; QAR per player (digits string ‚Äî typing-only `type="text"` inputMode numeric, ¬ß5) |
| `players` / `filled` | **total** spots / taken (host = 1). The create/edit form's slider chooses **players needed** (others, range 1‚Äì7); `total_spots = needed + 1` (host occupies the extra spot), so a card reads "{needed} spots left" right after create (¬ß5) |
| `skillMin` / `skillMax` | skill range over the 7-tier ladder (`SkillTier`; `skillMin === skillMax` ‚áí single level) |
| `waitlistOpen` / `waitlistSize` | queue toggle + 1‚Äì8 cap |
| `description` | free text (‚â§240) |
| `venueName, area, setting, court` | court-sport location |
| `isRoute, routeEnd, loop, km` | running route instead of a court |

- **Skill ladder = one ordered 7-tier enum** (`SKILL_TIERS` in `lib/types.ts`, lowest‚Üíhighest): `baby ¬∑ beginner ¬∑ low_int ¬∑ int ¬∑ high_int ¬∑ advance ¬∑ pro` (abbreviated labels `Baby ¬∑ Beginner ¬∑ Low Int ¬∑ Int ¬∑ High Int ¬∑ Advance ¬∑ Pro` via `format.skillTierLabel`, full single-level labels ‚Äî `Intermediate`/`Low Intermediate`/`High Intermediate` ‚Äî via `format.skillTierLabelFull`; **a displayed skill value goes through `format.formatSkillLevel`**: single = full word, range = abbreviated endpoints; `SkillLevel = SkillTier | 'any'` for the profile/filter context). `skillOrd`/`skillInRange` do the range math. The create/edit forms set the match's `skill_min`/`skill_max` with the shared **tap-only `SkillRangeSlider`** (¬ß5); profile (`users.skill_level`) + the sign-up questionnaire use the same tiers as a single value.
- **Venues:** `VENUES` seed (~27 curated Doha venues, `venuesForSport(sport)`). `VenueSelection` (`VenuePicker`) carries `id|'custom'|'route'`, name/area/setting, optional **court # (separate free-text)**, or route `endName/loop/km`. Running has no fixed venues ‚Üí start/finish route. (Note: the brief's `locDist` is **not** modelled.)

## 5. Core logic & conventions

- **Create vs Edit (`mode` prop).** Both `CreateMatchScreen` and `EditMatchScreen` take `mode: 'create'|'edit'`.
  - Edit pre-fills from the existing match; footer = **Save changes** + **Cancel match** (destructive confirm). Create opens blank; footer primary = **Create match**.
  - `CreateMatchScreen` (store flow) enforces a **required-choice guard**: Cost *and* Joining must be explicitly chosen before create (reminder sheet otherwise). In edit they're already set.
  - `EditMatchScreen` (demo flow) branches on `isCreate`: create shows the "New match ¬∑ {date}" eyebrow + "Host a match." hero and hides the destructive Cancel link; edit shows "You're hosting" + "Edit your match." + Cancel.
- **Persistence / source of truth.** The **demo** create/edit flow (`EditMatchScreen`) is the single source of truth for *the host's own match*: it writes **`localStorage` `connect:hostedMatch`** via `writeHostedMatch`, and `HomeScreen` reads it with the reactive `useHostedMatch()` ‚Äî so Home's "You're hosting" card reflects creates/edits instantly and **survives reload**. Save ‚Üí `/home`; Cancel ‚Üí `clearHostedMatch()` (card disappears). When present it is **preferred over** the seeded store hosted matches. The **store** flow (`CreateMatchScreen` + `actions.createMatch/updateMatch`) is the older, separate path ‚Äî see ¬ß7.
- **Fresh account = clean slate.** `actions.startFreshAccount()` (the onboarding "Create account" step) resets the seeded data via `freshDB()` **and** calls `clearHostedMatch()` ‚Äî otherwise a `connect:hostedMatch` left in localStorage from a prior session would make `HomeScreen` render the "You're hosting" variant instead of `FirstTimerHome`. So a brand-new account always lands on the first-timer Home (seeded Discover feed + Find a match / Host one).
- **Onboarding identity ‚Üí every screen.** Sign-up answers live in the `onboarding` proxy (`lib/onboarding.ts`, localStorage `connect.onboarding`): **`name`** is written on the Sign Up screen's *Create account* (after `resetOnboarding()`; empty falls back gracefully), **`sport`** and **`skill`** by the questionnaire steps. The single source for the *current user's display identity* is the store user `getUser(db, currentUserId)` ‚Äî **no screen reads `useAuth().user` directly**. For a **fresh account** (`isFreshAccount()`), `withSignedIn` overlays the onboarding `name`/`sport`/`skill_level` onto that user, so the sign-up answers win over the dev/mock signed-in profile. This matters because `AuthProvider` re-applies the dev `mockUser` ("Jen") via `setSignedInUser` on every mount/reload; without the overlay it would clobber the typed name. Result: the typed name **and the avatar initials derived from it** are consistent on every page (Home greeting/avatar, Profile, Edit Profile, Chat, "Hosted by", `MatchCard`/`AvatarStack`, post-match) and survive reload. `freshDB()` applies the same overlay for the module-init case (before `AuthProvider` mounts). The `mockUser` "Jen / padel / int" only shows in the seeded/demo (non-fresh) account.
- **First-timer Home feed = Discover, filtered.** `FirstTimerHome` renders the first 3 of the same `discoverFeed(db)` Discover uses (no separate array), filtered by the user's picks: `m.sport === sport && (skill === 'any' || skillInRange(skill, m.skill_min, m.skill_max))`. If that's empty it falls back to the full nearby feed with an honest label (`No {Sport} at your level ‚Äî nearby`) ‚Äî never blank (¬ß5). Ids stay stable across Home/Discover. **The Find a match / Host one buttons + this seeded rail live ONLY on `FirstTimerHome`** (the no-joined/no-created empty state); the returning-user `HomeScreen` deliberately omits them and shows Next up ¬∑ You're hosting ¬∑ My Matches instead. **Host entry points are mode-aware via the single `HOST_CREATE_ROUTE` const (`lib/hostedMatch.ts`)** ‚Äî every "host a match" button (FirstTimerHome "Host one", Discover header **+** and footer nudge) routes through it: **live** ‚Üí the **store** flow `/matches/create` (`actions.createMatch` RPC ‚Üí persists ‚Üí surfaces under "You're hosting" via `hostedMatches(db)`); **mock** ‚Üí the demo flow `/matches/create-demo` (writes `connect:hostedMatch`, read by `useHostedMatch()`). This is required because live Home **ignores** `connect:hostedMatch` (browser-global key, not per-user ‚Üí cross-account bleed), so a demo-created match would never show for a live user. Both flows route to `/home` on create.
- **Time handling ‚Äî 24h everywhere, no AM/PM.** Times are stored and displayed as **24h `"HH:mm"`**; there is **no 12h/AM-PM anywhere** in the UI. `format.to24h()` normalises an `"HH:mm"` string to zero-padded 24h for display; `MatchCard`/details use the existing 24h `hm`/`timeRange`. Start/End on **both** create/edit forms are **typeable** ‚Äî the shared `WhenCard.TimeField` renders **two digit-only cells (HH / MM) with a fixed `:` between them** (no clock glyph); each cell accepts digits and the pair commits a value normalised by `datetime.normalizeTime` on blur/Enter (clamped to 0‚Äì23 : 0‚Äì59, falling back to the prior value on garbage).
- **Date selection.** `WhenCard` shows a 10-day quick-pick strip **anchored to today** plus a native `<input type="date">` calendar pill for any date; picking a date outside the window re-anchors the strip so the selection stays visible. `dateLabel` is derived from `dateKey` via `datetime.labelFromKey` (correct for any month). Create flows pass `restrictPast` so the calendar's `min` is today.
- **Numbers-only inputs.** Price uses `type="number" inputMode="numeric"`, `onChange` sanitises to `/[^0-9]/g`, keydown blocks `e/E/+/-/.`, and the **`.num`** class hides native spinners.
- **Scroll model.** One scroll owner per screen: `Shell`/`.screen` clips with `overflow:hidden`; an inner `.scroll` owns vertical scroll (`56px 22px 130px` padding for back-button + sticky footer). Horizontal rails (date strip, sport pills) use **`.hscroll`** (hidden scrollbar). Never `scrollIntoView`.
- **Duration logic** (in `WhenCard`, shared by both forms). **Field order is Date ‚Üí Duration ‚Üí Start ‚Üí End.** `duration` (minutes) is the source of truth ‚Äî the host picks it first (presets `30/60/90/120` or the **typing-only** custom popover: `type="text"` + `inputMode="numeric"` + digit sanitise + `.num`, never `type="number"`). **End is read-derived, never typed** (`EndField`, shows an `AUTO` tag): `End = Start + duration` via `datetime.addMinutes`, whose `%1440` wrap handles the **day-rollover** edge case cleanly (e.g. 23:30 + 90m ‚Üí 01:00). Changing **either** Start or Duration re-rolls End. `duration` lives in the **shared draft** (`usePersistedState`, e.g. `match:create:duration`) ‚Äî passed in via `duration`/`onDuration`, not WhenCard-local state ‚Äî so it persists across the flow and the `start_time`/`end_time` written in `save()` can't drift; create defaults to 90, edit pre-fills via `diffMinutes(existing start, end)`.
- **Spots = "players needed", host occupies one.** The create/edit form's Players slider chooses how many **others** the host needs (range **1‚Äì7**, labelled "Players needed ¬∑ Not counting you"); `total_spots = needed + 1`, `spots_available = needed` at create (RPC/mock both seat the host). So the card reads "{needed} spots left" immediately. `matchKind` still derives Singles (total 2) / Doubles (total 4) from `total_spots`, so the 2‚Äì8 total range is unchanged. **When editing the player count**, `CreateMatchScreen.save()` recomputes `spots_available = max(newTotal ‚àí alreadyJoined, 0)` (joined = `existing.total_spots ‚àí existing.spots_available`) so the count can't drift; `updateMatch`'s editable cols include `spots_available`.
- **Price input is typing-only** ‚Äî `type="text"` + `inputMode="numeric"`, `onChange` sanitises to `/[^0-9]/g`, `.num` class (no spinners). Don't use `type="number"` for price (it shows steppers).
- **`MatchCard` always shows the host.** Pass `host` to the card (incl. the Home "You're hosting" card ‚Üí `host={getUser(db, m.host_id)}`) so the "Hosted by ‚Ä¶" line renders.
