# CLAUDE.md — Connect! (app/)

Implementation memory for the **runnable app** in `app/`. Read this before editing.
Product/business rules and the Supabase target schema live in the spec-level
`../Connect! UI Lab/CLAUDE.md` (referred to here as **spec §x**); this file documents
what's *actually built in code*. When code and a doc disagree, trust the code and flag it.

---

## 1. What the app is

- **Connect!** — a mobile-first pickup-sports matchmaking app for **Doha, Qatar** (currency **QAR**).
- Sports: **padel · tennis · badminton · running**.
- Core user jobs: **discover** open matches → **host** a match → **join / request / approve** players (+ FIFO **waitlist**) → **chat** (per-match group + DMs) → **record results** post-match.
- Stage 1 scope: free matching only, cash-on-arrival (no in-app payments), 18+, Doha-only. See spec §1.

## 2. Tech & file structure

**Stack** — React 19 + TypeScript, **Vite 8**, **Tailwind v4** (`@tailwindcss/vite`, config-less — tokens live in `src/index.css`), `react-router-dom` v7, **lucide-react** icons, `clsx`/`tailwind-merge`/`cva` (via `cn()` in `lib/utils.ts`). No shadcn; components are local. Run: `npm run dev` (Vite), `npm run build` (`tsc -b && vite build`), `npm run lint`.

**Entry / frame** — `index.html` (loads Inter + Instrument Serif + IBM Plex Sans Arabic) → `main.tsx` → `App.tsx` (`<Routes>`). Every screen renders inside `Shell` = the phone frame: `max-w-[430px]`, `h-dvh`, cream page on a dark backdrop, ambient `Blobs`, optional floating `BottomNav`.

**State** — single **in-memory reactive store** (`lib/store.ts`) via `useSyncExternalStore`, seeded from `lib/mock/*`. Repository-shaped accessors mirror future Supabase queries. A few things persist to `localStorage`: `connect.freshAccount`, profile-done flag, and **`connect:hostedMatch`** (see §5). ES modules throughout — **no `window` globals, no inline Babel** (that was the old prototype in `../Connect! UI Lab/`).

## 4. Data model

There are **two shapes**: the canonical store `Match` (Supabase-shaped, snake_case) and the **edit-form / `HostedMatch`** shape (camelCase, what the create/edit form + Home card use). Keep them in sync via the mapper in `EditMatchScreen.save()`.

**Store `Match`** (`lib/types.ts`) — `id, host_id, sport, name?, venue_id?, venue_name, venue_location?, court_number?, route_start?, route_end?, round_trip, start_time(ISO), end_time(ISO), skill_min, skill_max, total_spots, spots_available, fee_total?, fee_per_player?, join_mode('open'|'approval'|'invite'), match_type?('casual'|'competition'), gender_restriction?('mixed'|'ladies'), status('active'|'cancelled'), notes?, created_at`. **`gender_restriction`** ('ladies' = female players only; absent ⇒ 'mixed') is **server-enforced** — the gate lives in RLS + the join/request/invite/waitlist RPCs + DB triggers (see the changelog/§5), NOT just the UI. The create/edit "Open to" toggle is shown ONLY to female hosts; `genderBlocks(d, matchId)` (store) mirrors the gate so the UI never offers a CTA the server will reject, and the `MatchCard genderBlocked` prop renders a disabled "Ladies only" state for a barred (male) viewer (label `match.gender.womenOnly`, retitled "Ladies only"). The `LadiesBadge` (one component) renders on both card variants + Match Details. **`match_type`** ('casual' default / 'competition') is captured on the create form and persisted in the **mock store only** (no live column yet — `repo.mapMatch` defaults it to 'casual'); it drives the Profile "Recent matches" pill — casual & running show "Played", only competition (non-running) matches with a result surface Won/Lost (§5). **Skill is a RANGE** over the 7-tier ladder (`skill_min`/`skill_max`, both `SkillTier`; `skill_min === skill_max` ⇒ a single level). DB CHECK enforces `skill_ord(skill_min) ≤ skill_ord(skill_max)`; `create_match` takes `p_skill_min`/`p_skill_max`. Read-time `MatchStatus` is computed, not stored (§5). **`name`** is the optional host-given title from the create/edit form: persisted live via the `matches.name` column (`create_match`'s `p_name` + `updateMatch` cols + `repo.mapMatch`), and rendered as the **`MatchCard` title** when present (else the card falls back to the derived `matchKind at courtLabel`, e.g. "Doubles at 4"). So every `MatchCard` surface (Home "You're hosting", Discover, All Matches) shows the typed name; venue still shows on the card art + the MapPin line.

**`HostedMatch`** (`lib/hostedMatch.ts`) — the host's own match, the form's working shape:

| Field | Meaning |
|---|---|
| `sport` | padel / tennis / badminton / running |
| `name` | match title (serif headline) |
| `dateKey` / `dateLabel` | ISO `yyyy-mm-dd` / display `"Thu · May 23"` (derived from `dateKey` via `labelFromKey`, any month) |
| `startTime` / `endTime` | **24h `"HH:mm"`**, displayed as-is (no AM/PM anywhere — see §5) |
| `matchType` | `casual` \| `competition` |
| `gender` | `mixed` \| `ladies` |
| `requireApproval` | join instantly vs approve (maps to `join_mode`) |
| `isFree` / `pricePerPlayer` | Free vs split; QAR per player (digits string — typing-only `type="text"` inputMode numeric, §5) |
| `players` / `filled` | **total** spots / taken (host = 1). The create/edit form's slider chooses **players needed** (others, range 1–7); `total_spots = needed + 1` (host occupies the extra spot), so a card reads "{needed} spots left" right after create (§5) |
| `skillMin` / `skillMax` | skill range over the 7-tier ladder (`SkillTier`; `skillMin === skillMax` ⇒ single level) |
| `waitlistOpen` / `waitlistSize` | queue toggle + 1–8 cap |
| `description` | free text (≤240) |
| `venueName, area, setting, court` | court-sport location |
| `isRoute, routeEnd, loop, km` | running route instead of a court |

- **Skill ladder = one ordered 7-tier enum** (`SKILL_TIERS` in `lib/types.ts`, lowest→highest): `baby · beginner · low_int · int · high_int · advance · pro` (abbreviated labels `Baby · Beginner · Low Int · Int · High Int · Advance · Pro` via `format.skillTierLabel`, full single-level labels — `Intermediate`/`Low Intermediate`/`High Intermediate` — via `format.skillTierLabelFull`; **a displayed skill value goes through `format.formatSkillLevel`**: single = full word, range = abbreviated endpoints; `SkillLevel = SkillTier | 'any'` for the profile/filter context). `skillOrd`/`skillInRange` do the range math. The create/edit forms set the match's `skill_min`/`skill_max` with the shared **tap-only `SkillRangeSlider`** (§5); profile (`users.skill_level`) + the sign-up questionnaire use the same tiers as a single value.
- **Venues:** `VENUES` seed (~27 curated Doha venues, `venuesForSport(sport)`). `VenueSelection` (`VenuePicker`) carries `id|'custom'|'route'`, name/area/setting, optional **court # (separate free-text)**, or route `endName/loop/km`. Running has no fixed venues → start/finish route. (Note: the brief's `locDist` is **not** modelled.)

## 5. Core logic & conventions

- **Create vs Edit (`mode` prop).** Both `CreateMatchScreen` and `EditMatchScreen` take `mode: 'create'|'edit'`.
  - Edit pre-fills from the existing match; footer = **Save changes** + **Cancel match** (destructive confirm). Create opens blank; footer primary = **Create match**.
  - `CreateMatchScreen` (store flow) enforces a **required-choice guard**: Cost *and* Joining must be explicitly chosen before create (reminder sheet otherwise). In edit they're already set.
  - `EditMatchScreen` (demo flow) branches on `isCreate`: create shows the "New match · {date}" eyebrow + "Host a match." hero and hides the destructive Cancel link; edit shows "You're hosting" + "Edit your match." + Cancel.
- **Persistence / source of truth.** The **demo** create/edit flow (`EditMatchScreen`) is the single source of truth for *the host's own match*: it writes **`localStorage` `connect:hostedMatch`** via `writeHostedMatch`, and `HomeScreen` reads it with the reactive `useHostedMatch()` — so Home's "You're hosting" card reflects creates/edits instantly and **survives reload**. Save → `/home`; Cancel → `clearHostedMatch()` (card disappears). When present it is **preferred over** the seeded store hosted matches. The **store** flow (`CreateMatchScreen` + `actions.createMatch/updateMatch`) is the older, separate path — see §7.
- **Fresh account = clean slate.** `actions.startFreshAccount()` (the onboarding "Create account" step) resets the seeded data via `freshDB()` **and** calls `clearHostedMatch()` — otherwise a `connect:hostedMatch` left in localStorage from a prior session would make `HomeScreen` render the "You're hosting" variant instead of `FirstTimerHome`. So a brand-new account always lands on the first-timer Home (seeded Discover feed + Find a match / Host one).
- **Onboarding identity → every screen.** Sign-up answers live in the `onboarding` proxy (`lib/onboarding.ts`, localStorage `connect.onboarding`): **`name`** is written on the Sign Up screen's *Create account* (after `resetOnboarding()`; empty falls back gracefully), **`sport`** and **`skill`** by the questionnaire steps. The single source for the *current user's display identity* is the store user `getUser(db, currentUserId)` — **no screen reads `useAuth().user` directly**. For a **fresh account** (`isFreshAccount()`), `withSignedIn` overlays the onboarding `name`/`sport`/`skill_level` onto that user, so the sign-up answers win over the dev/mock signed-in profile. This matters because `AuthProvider` re-applies the dev `mockUser` ("Jen") via `setSignedInUser` on every mount/reload; without the overlay it would clobber the typed name. Result: the typed name **and the avatar initials derived from it** are consistent on every page (Home greeting/avatar, Profile, Edit Profile, Chat, "Hosted by", `MatchCard`/`AvatarStack`, post-match) and survive reload. `freshDB()` applies the same overlay for the module-init case (before `AuthProvider` mounts). The `mockUser` "Jen / padel / int" only shows in the seeded/demo (non-fresh) account.
- **First-timer Home feed = Discover, filtered.** `FirstTimerHome` renders the first 3 of the same `discoverFeed(db)` Discover uses (no separate array), filtered by the user's picks: `m.sport === sport && (skill === 'any' || skillInRange(skill, m.skill_min, m.skill_max))`. If that's empty it falls back to the full nearby feed with an honest label (`No {Sport} at your level — nearby`) — never blank (§5). Ids stay stable across Home/Discover. **The Find a match / Host one buttons + this seeded rail live ONLY on `FirstTimerHome`** (the no-joined/no-created empty state); the returning-user `HomeScreen` deliberately omits them and shows Next up · You're hosting · My Matches instead. **Host entry points are mode-aware via the single `HOST_CREATE_ROUTE` const (`lib/hostedMatch.ts`)** — every "host a match" button (FirstTimerHome "Host one", Discover header **+** and footer nudge) routes through it: **live** → the **store** flow `/matches/create` (`actions.createMatch` RPC → persists → surfaces under "You're hosting" via `hostedMatches(db)`); **mock** → the demo flow `/matches/create-demo` (writes `connect:hostedMatch`, read by `useHostedMatch()`). This is required because live Home **ignores** `connect:hostedMatch` (browser-global key, not per-user → cross-account bleed), so a demo-created match would never show for a live user. Both flows route to `/home` on create.
- **Time handling — 24h everywhere, no AM/PM.** Times are stored and displayed as **24h `"HH:mm"`**; there is **no 12h/AM-PM anywhere** in the UI. `format.to24h()` normalises an `"HH:mm"` string to zero-padded 24h for display; `MatchCard`/details use the existing 24h `hm`/`timeRange`. Start/End on **both** create/edit forms are **typeable** — the shared `WhenCard.TimeField` renders **two digit-only cells (HH / MM) with a fixed `:` between them** (no clock glyph); each cell accepts digits and the pair commits a value normalised by `datetime.normalizeTime` on blur/Enter (clamped to 0–23 : 0–59, falling back to the prior value on garbage).
- **Date selection.** `WhenCard` shows a 10-day quick-pick strip **anchored to today** plus a native `<input type="date">` calendar pill for any date; picking a date outside the window re-anchors the strip so the selection stays visible. `dateLabel` is derived from `dateKey` via `datetime.labelFromKey` (correct for any month). Create flows pass `restrictPast` so the calendar's `min` is today.
- **Numbers-only inputs.** Price uses `type="number" inputMode="numeric"`, `onChange` sanitises to `/[^0-9]/g`, keydown blocks `e/E/+/-/.`, and the **`.num`** class hides native spinners.
- **Scroll model.** One scroll owner per screen: `Shell`/`.screen` clips with `overflow:hidden`; an inner `.scroll` owns vertical scroll (`56px 22px 130px` padding for back-button + sticky footer). Horizontal rails (date strip, sport pills) use **`.hscroll`** (hidden scrollbar). Never `scrollIntoView`.
- **Duration logic** (in `WhenCard`, shared by both forms). **Field order is Date → Duration → Start → End.** `duration` (minutes) is the source of truth — the host picks it first (presets `30/60/90/120` or the **typing-only** custom popover: `type="text"` + `inputMode="numeric"` + digit sanitise + `.num`, never `type="number"`). **End is read-derived, never typed** (`EndField`, shows an `AUTO` tag): `End = Start + duration` via `datetime.addMinutes`, whose `%1440` wrap handles the **day-rollover** edge case cleanly (e.g. 23:30 + 90m → 01:00). Changing **either** Start or Duration re-rolls End. `duration` lives in the **shared draft** (`usePersistedState`, e.g. `match:create:duration`) — passed in via `duration`/`onDuration`, not WhenCard-local state — so it persists across the flow and the `start_time`/`end_time` written in `save()` can't drift; create defaults to 90, edit pre-fills via `diffMinutes(existing start, end)`.
- **Spots = "players needed", host occupies one.** The create/edit form's Players slider chooses how many **others** the host needs (range **1–7**, labelled "Players needed · Not counting you"); `total_spots = needed + 1`, `spots_available = needed` at create (RPC/mock both seat the host). So the card reads "{needed} spots left" immediately. `matchKind` still derives Singles (total 2) / Doubles (total 4) from `total_spots`, so the 2–8 total range is unchanged. **When editing the player count**, `CreateMatchScreen.save()` recomputes `spots_available = max(newTotal − alreadyJoined, 0)` (joined = `existing.total_spots − existing.spots_available`) so the count can't drift; `updateMatch`'s editable cols include `spots_available`.
- **Price input is typing-only** — `type="text"` + `inputMode="numeric"`, `onChange` sanitises to `/[^0-9]/g`, `.num` class (no spinners). Don't use `type="number"` for price (it shows steppers).
- **`MatchCard` always shows the host.** Pass `host` to the card (incl. the Home "You're hosting" card → `host={getUser(db, m.host_id)}`) so the "Hosted by …" line renders. The avatar stack builds a `roster` that **prepends the host if absent from `players`**, so the host monogram heads the stack even before `match_players` hydrates.
- **Lifecycle / status** (`computeStatus`, read-time, no cron): `open` (spots left) → `full` → `live` (start≤now<end) → `completed` (≤24h after end) → `closed`; `cancelled` is stored. Cancel cleanly only **≥2h** before start (`canCancelCleanly`), else no-show. **`actions.cancelMatch` optimistically mutates the store in BOTH modes** (sets `status:'cancelled'`, expires waitlist) then, in live mode, `afterWrite`s the Supabase update + rehydrates — so Home (which navigates immediately after cancel) drops the card on the next render instead of waiting for the round-trip. **Cancelling also removes the match's group chat from the Chat page:** `inboxThreads` filters out any thread whose match is `cancelled`, so the conversation disappears from the inbox for every participant on the next render (live: the status flip propagates via Realtime). The thread row is **not** deleted — the read-only `MatchChatScreen` ("This match was cancelled — chat is read-only") stays reachable by deep-link as a graceful fallback so history isn't lost.
- **Attend check-in (NEXT UP)** (`status.attendCheckInOpen`, `store.checkIn`/`hasCheckedIn`, read-time, **no cron/trigger**): the NEXT UP card's **Attend** button is a low-friction **POSITIVE presence signal** — it records `match_players.attended = true` for the tapping player **only** (self-only; the live write is a direct `match_players.update` RLS-guarded by **`mp_update_self`** `player_id = auth.uid()`, migration `connect_attend_checkin_rls`) and **never marks a no-show** (un-check returns to `null`, never `false`; `false` stays reserved for the post-match flag). Visibility is read-time from `start_time`: **opens ~30 min before start, stays open through `live` + just-played (`completed` ≤24h), never earlier** (no meaningless early taps) — `HomeScreen` gates the card with `action={attendCheckInOpen ? 'attend' : 'view'}` (the `MatchCard` `attend` branch now sits ABOVE `lifecycleHasAction` so it's honored in `live`/`completed` too; only NEXT UP passes `action='attend'`). A check-in **pre-fills "Played"** in post-match step 1 (`RecordResultForm` reads `attended === true` → not in `absent`, and shows a "Checked in" tag) — confirm/flag exceptions only. It carries **NO score**: win/lose/draw (step 2) is untouched and remains the ONLY source of win rate. The no-show corroboration path (`noShowReportThreshold` + `no_show_reports`) stays the source of truth when the signal and reality diverge. **Reach across the timeline:** `upcomingJoinedMatches` includes `live` (not just `open`/`full`) so a joined match stays on NEXT UP through kickoff — attend is reachable pre-start **and** `live`; at `end_time` the match leaves NEXT UP and the transient **JUST PLAYED** card takes over with **Record results** (the post-match flow; attendance pre-filled from check-ins) — no standalone attend button in the just-played state (recording covers attendance). Selectors: `upcomingJoinedMatches` (NEXT UP, now open/full/live) + `justPlayedMatches` (completed ≤24h, awaiting your result); the `RecordResultHomeCard` is wired at the top of `HomeScreen` (spec §4).
- **Waitlist** (`actions.joinWaitlist/leaveWaitlist`, `waitlistPosition`): full matches only, FIFO by `created_at`, **auto-promote** the earliest waitlister when a slot frees — **no host approval on promotion** even for `approval` matches. See spec §5 for the full rules (threshold table in `noShowReportThreshold`). **Waitlisted matches fold into Home "My Matches"** (alongside joined + requested) via the `waitlistedMatches(d)` selector → `thisWeekMatches`, rendered with a passive **"On waitlist · #N"** badge (`MatchCard joinStatus='waitlisted'` + `waitlistPosition`; leave from Chat Alerts / Match Details). They drop out on leave (off `myWaitlistEntries`), auto-promotion (→ joined, badge clears), or cancel/start (read-time) — mirroring how a declined request leaves "My Matches".
